🤖 ใช้ Agent ให้เป็น ใช้ให้คุ้ม

คู่มือฉบับเข้าใจง่าย สำหรับ 3 agent ที่สร้างไว้ในโปรเจ็คนี้

1 Agent คืออะไร (เข้าใจแบบง่ายที่สุด)

นึกภาพว่า Claude เป็นพนักงาน 1 คนที่ ความจำสั้นมาก — ทุกครั้งที่เริ่มคุยใหม่ ต้องอ่านทุกอย่างใหม่หมด

Agent คือการจ้างพนักงานเฉพาะทางเพิ่ม ที่มี "สมุดจดกฎประจำตัว" (ไฟล์ .md) ติดตัวมาตลอด พอเรียกใช้ครั้งไหน มันจำกฎเฉพาะด้านได้ทันที ไม่ต้องอธิบายซ้ำ

ในโปรเจ็คนี้มี 3 คน: 🧪 qa-tester (ทดสอบเว็บ), 🗄️ db-specialist (งานฐานข้อมูล), 🎨 ui-builder (สร้าง/แก้ UI)

2 กฎเหล็กข้อเดียวที่ต้องจำ: Agent ไม่รู้ว่าเราคุยอะไรกันมา

Agent เริ่มต้นด้วยความจำเป็นศูนย์ — มันไม่เห็นบทสนทนาที่กำลังคุยอยู่นี้เลย รู้แค่ 2 อย่าง: (1) สิ่งที่เขียนไว้ในไฟล์ของมันเอง (2) คำสั่งที่พิมพ์ตอนเรียกครั้งนั้น

❌ สั่งสั้นเกินไป Agent จะไม่รู้ว่า "ที่เพิ่งแก้" คืออะไร ไฟล์ไหน เปลี่ยนอะไรมา "ให้ qa-tester เทสหน่อย"
✅ ให้บริบทครบ บอกหน้า/ไฟล์ + สิ่งที่เพิ่งเปลี่ยน + อยากได้ผลแบบไหน "ทดสอบหน้า /orders/new โฟกัสปุ่มลงย้อนหลังกับ channel ที่เพิ่งเปลี่ยนเป็นชุด TikTok/FB โปรไฟล์ ทดสอบมือถือ+desktop แล้วลบ test data ทิ้งด้วย"
สูตรจำง่าย 3 ข้อ: (1) งานคืออะไร/ไฟล์ไหน (2) เพิ่งมีอะไรเปลี่ยน (3) อยากได้ผลลัพธ์แบบไหน

3 ตัวอย่างสั่งงานจริง — ทีละตัว

🧪
qa-tester
ทดสอบเว็บจริงผ่าน browser ก่อน commit
ให้ qa-tester ทดสอบหน้า /tasks ว่า modal ตั้งงาน 9 ประเภทที่เพิ่งเพิ่มใหม่ (ทวงชำระ/ขอรีวิว/เสนอโปร/แนะนำวิธีกิน ฯลฯ) เปิดได้ครบทุกประเภท กดบันทึกงานได้จริง ทดสอบบนมือถือ 390px ก่อนเป็นหลัก แล้วค่อยเช็ค desktop
ทำไมสั่งแบบนี้: บอกฟีเจอร์ชัด (9 ประเภทใหม่) + ลำดับความสำคัญ (มือถือก่อน) — agent ไม่ต้องเดาว่าจะเทสอะไรก่อน
🗄️
db-specialist
งาน schema / backfill / query วิเคราะห์ข้อมูล
ให้ db-specialist เช็คว่าตอนนี้มีเบอร์โทรลูกค้ารูปแบบเพี้ยน (มีขีด/เว้นวรรค/คำนำหน้า) หลงเหลืออยู่อีกไหม หลังจาก backfill ไปรอบก่อน — ถ้าเจอให้รายงานจำนวนก่อน อย่าเพิ่งแก้อะไร
ทำไมสั่งแบบนี้: ระบุ "อย่าเพิ่งแก้" ชัดเจน (dry-run) — งาน DB ต้องขอ approve ก่อนเขียนจริงเสมอ
🎨
ui-builder
สร้าง/แก้หน้า UI ตาม design system
ให้ ui-builder เพิ่มปุ่ม "ส่งออก Excel" ในหน้า /orders ตรง filter bar ใช้ style เดียวกับปุ่ม "กรอกออเดอร์ใหม่" ที่มีอยู่แล้ว ต้องใช้งานได้ดีบนมือถือด้วย
ทำไมสั่งแบบนี้: ชี้ตำแหน่งชัด + อ้างอิงของเดิมที่มีอยู่ (ให้ agent เลียนแบบ pattern แทนคิดใหม่)

4 รัน Agent หลายตัวพร้อมกัน (Parallel)

ถ้างาน 2 อย่างไม่ต้องรอกัน (คนละไฟล์ คนละเรื่อง) สั่งพร้อมกันได้ในข้อความเดียว — เร็วกว่ารอทีละคน

ตัวอย่างจากงานจริงที่ทำได้:
🎨 ui-builder
เพิ่มหน้า export UI
‖
🗄️ db-specialist
เตรียม query ยอดขายสำหรับ export

ทั้งสองงานไม่ได้รอผลกันและกัน → สั่งพร้อมกันได้ ผมจะเรียก Agent tool 2 ครั้งในคำตอบเดียว

⚠️ เมื่อไหร่ห้าม parallel: ถ้างาน B ต้องใช้ผลจากงาน A (เช่น "เช็คว่า schema พอไหมก่อน แล้วค่อยสร้างฟอร์ม") ต้องรันเรียงกัน ไม่ใช่พร้อมกัน — บอกผมได้เลยว่า "งานนี้ต้องรอกันไหม" ถ้าไม่แน่ใจ

5 Background Agent (รันเบื้องหลัง)

งานที่ ใช้เวลานาน (เทสยาว ๆ, audit ใหญ่ ๆ) สั่งให้รันเบื้องหลังได้ — คุณคุยงานอื่นกับผมต่อไปพลางระหว่างรอ พอเสร็จจะมีแจ้งเตือนอัตโนมัติ

ตัวอย่างที่เหมาะกับ background:
  • "ให้ qa-tester ทดสอบทุกหน้าในระบบแบบละเอียด รันเบื้องหลังได้เลย ระหว่างนี้เรามาคุยเรื่องฟีเจอร์ถัดไปกัน"
  • "ให้ db-specialist ไล่เช็คข้อมูลทั้ง 966 ออเดอร์หาความผิดปกติ ใช้เวลานานได้ ไม่ต้องรีบ"

ไม่เหมาะกับ background: งานที่ต้องรู้ผลก่อนถึงจะสั่งงานต่อได้ (เช่น "เช็คก่อนว่าแก้ได้ไหม แล้วฉันจะบอกว่าทำต่อยังไง")

6 สรุปเปรียบเทียบ: เมื่อไหร่ใช้ Agent vs ให้ Claude หลักทำเอง

สถานการณ์ใช้ Agentให้ทำเองในแชทนี้
งานสั้น ต่อเนื่องจากที่คุยอยู่✅
งานแยกชัดเจน มีขอบเขตจบในตัว✅
อยากประหยัด context ของบทสนทนาหลัก✅
ต้องตัดสินใจต่อเนื่องแบบ real-time✅
งานใช้เวลานาน อยากทำอย่างอื่นคู่กัน✅ (background)

7 สรุปสั้น ๆ ก่อนจบ

1. เรียกชื่อ agent ตรง ๆ ในคำสั่ง เช่น "ให้ qa-tester...", "ให้ db-specialist..."
2. ให้บริบทเสมอ — ห้ามคิดว่า agent รู้เหมือนที่เราคุยกันมา
3. งานไม่พึ่งพากัน → สั่งพร้อมกันได้ (parallel)
4. งานยาว/ไม่รีบ → สั่งเบื้องหลังได้ (background)
5. ไม่แน่ใจว่าควรใช้ agent ไหม/สั่งแบบไหน → ถามผมได้เสมอ ผมช่วยแปลคำสั่งให้
ไฟล์นี้เป็นคู่มือส่วนตัว ไม่ใช่โค้ดแอป — ลบทิ้งได้ตลอดเวลาถ้าไม่ต้องการแล้ว
